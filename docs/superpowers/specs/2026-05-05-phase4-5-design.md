# 浙大食堂助手 Phase 4-5 设计文档

## 概述

本文档定义 Phase 4（功能完善 + 数据扩充）和 Phase 5（比赛材料）的实现设计。

**目标：**
- 优化 SVG 占位图，提升视觉辨识度
- 升级 AI 推荐算法，引入多因素加权打分
- 增强对话交互，实现主动追问引导
- 制作演示视频，展示三大创新点

---

## Phase 4A：SVG 占位图优化

### 当前问题

现有 SVG 结构统一但缺乏辨识度：
- 所有菜品使用同一个 🍽️ emoji
- 仅背景色不同，用户无法快速区分菜品
- 缺少关键信息展示（价格、卡路里、标签）

### 设计方案

**卡片结构：**
```
┌─────────────────────┐
│  渐变背景（按口味）   │
│  ┌───────────────┐  │
│  │   食物 emoji   │  │
│  │     🍜        │  │
│  └───────────────┘  │
│                     │
│   菜品名称           │
│   ¥12 · 450kcal     │
│   ┌─────────────┐   │
│   │ 辣 · 荤 · 人气│   │
│   └─────────────┘   │
│   高蛋白 · 增肌推荐  │
└─────────────────────┘
```

**显示信息：**
- 食物 emoji（对应菜品类型）
- 菜品名称
- 价格 + 卡路里
- 口味标签（辣/清淡/传统等）
- 荤素标识（荤/素/面食）
- 营养标签（高蛋白/低卡/增肌/减脂等）

**颜色映射规则：**

| 主标签 | 颜色系 | 渐变起止 |
|--------|--------|---------|
| 辣 | 红色系 | #FF6B6B → #EE5A5A |
| 清淡 | 绿色系 | #51CF66 → #40C057 |
| 传统/人气 | 橙色系 | #FF922B → #FD7E14 |
| 面食 | 黄色系 | #FCC419 → #F59F00 |
| 高蛋白 | 紫色系 | #9775FA → #845EF7 |
| 低卡 | 青色系 | #22D3EE → #06B6D4 |

**多标签处理：** 取第一个标签（tags[0]）决定主色。

**食物 emoji 映射：**

| 菜品类型 | Emoji |
|---------|-------|
| 面食类 | 🍜 |
| 米饭类 | 🍚 |
| 包子/饺子 | 🥟 |
| 肉类 | 🥩 |
| 鱼类 | 🐟 |
| 蔬菜类 | 🥬 |
| 汤类 | 🍲 |
| 炒菜类 | 🍳 |

### 实现要点

1. 生成 32 个新 SVG 文件，替换现有占位图
2. SVG 尺寸保持 400×300，适配现有卡片布局
3. 渐变背景使用 linearGradient 定义
4. 标签区域使用 flex 布局模拟（SVG 内用绝对定位）

## Phase 4B：AI 推荐算法升级

### 当前问题

现有算法是简单标签匹配：
```javascript
if (haystack.includes("辣") && text.includes("辣")) score += 3;
```
- 仅匹配关键词，无权重区分
- 未考虑营养目标、价格、多样性

### 设计方案

**推荐得分公式：**
```
总分 = 口味偏好×0.30 + 营养匹配×0.25 + 热度评分×0.20
      + 价格匹配×0.15 + 多样性×0.05 + 时间匹配×0.05
```

**各因素计算方法：**

| 因素 | 计算逻辑 |
|------|---------|
| 口味偏好 | 用户偏好标签与菜品标签匹配数 × 10 |
| 营养匹配 | 根据目标计算：减脂=低卡高分，增肌=高蛋白高分 |
| 热度评分 | (rating × 20) + review_count × 0.5 |
| 价格匹配 | 预算范围内满分，超出按比例扣分 |
| 多样性 | 最近 3 次推荐中未出现满分，出现扣分 |
| 时间匹配 | 早餐时段面食高分，晚餐时段正餐高分 |

**营养匹配细则：**

| 目标 | 高分条件 | 低分条件 |
|------|---------|---------|
| 减脂 | calories < 400, fat < 15 | calories > 600 |
| 增肌 | protein > 25, calories > 400 | protein < 15 |
| 均衡 | 各营养素接近平均值 | 极端值 |

**输出格式保持不变：**
```json
{
  "recommendations": [
    {"dish": {...}, "reason": "...", "match_score": 85}
  ]
}
```

### 实现要点

1. 后端 `services/recommendation.py` 重写 `ai_recommend` 函数
2. 前端 `Recommend.tsx` 的 `pickLocalRecommendations` 同步更新
3. 新增用户历史推荐记录（用于多样性计算）
4. API 响应增加 `factors_breakdown` 字段（可选，用于调试）

## Phase 4C：对话增强

### 当前问题

用户输入模糊时（如"推荐个菜"），AI 直接返回结果，未主动收集偏好信息。

### 设计方案

**触发条件：**
用户输入缺少以下任一信息时触发追问：
- 口味偏好（辣/清淡）
- 预算范围
- 营养目标（减脂/增肌/均衡）
- 荤素偏好

**追问流程：**

```
用户: "推荐个菜"
  ↓
AI 检测信息不足
  ↓
追问第一步：口味
  ┌─────────────────────┐
  │ 你偏好什么口味？     │
  │ [辣] [清淡] [随意]   │
  └─────────────────────┘
  ↓
用户选择: [辣]
  ↓
追问第二步：预算
  ┌─────────────────────┐
  │ 预算范围？           │
  │ [10元内] [15元内] [不在意] │
  └─────────────────────┘
  ↓
用户选择: [15元内]
  ↓
追问第三步：目标
  ┌─────────────────────┐
  │ 今天的目标？         │
  │ [减脂] [增肌] [均衡] [随意] │
  └─────────────────────┘
  ↓
用户选择: [减脂]
  ↓
信息收集完成，执行推荐算法
```

**UI 设计：**

选项按钮样式：
- 横向排列，3-4 个选项
- 圆角卡片，点击高亮
- 选中后自动发送并进入下一步

**实现要点：**

1. 前端新增 `QuickReplyButtons` 组件
2. AI 响应增加 `quick_reply` 字段：
```json
{
  "reply": "你偏好什么口味？",
  "quick_reply": {
    "step": "taste",
    "options": ["辣", "清淡", "随意"]
  }
}
```
3. 后端 `/api/chat` 增加追问逻辑
4. 用户选择存入对话上下文，最终汇总调用推荐算法

## Phase 5：比赛材料

### 演示视频

**时长：** 3 分钟

**结构：**

| 段落 | 时长 | 内容 | 形式 |
|------|------|------|------|
| 开场 | 20秒 | 标题动画 + 痛点文字 | HTML 动画 |
| 创新点 1 | 60秒 | AI 对话式推荐演示 | 录屏 |
| 创新点 2 | 40秒 | 偏好记忆系统 | 录屏 + 动画标注 |
| 创新点 3 | 40秒 | 营养目标驱动 | 录屏 + 动画标注 |
| 结尾 | 20秒 | Logo + 标语 | HTML 动画 |

**HTML 动画内容：**

1. **开场动画：**
   - 标题 "浙大食堂助手" 渐入
   - 问题文字依次出现："今天吃什么？" → "选择太多？" → "营养怎么平衡？"
   - 产品定位文字滑入

2. **创新点标注动画：**
   - 录屏暂停时，创新点文字从侧边滑入
   - 高亮区域框动画
   - 图标动画（对话气泡、大脑、营养图表）

3. **结尾动画：**
   - 三个创新点图标排列
   - Logo 渐入
   - 标语："让每一餐都有答案"

**技术实现：**
- 使用项目现有的 Motion (Framer Motion) 库
- 创建独立 HTML 页面 `demo-animation.html`
- 录制浏览器窗口（使用 OBS 或系统录屏）

### 说明文档 PDF

**内容结构：**

1. 产品概述（1页）
   - 定位、目标用户、核心价值

2. 用户洞察（1页）
   - 痛点分析、调研数据（如有）

3. 产品方案（2页）
   - 功能架构、页面流程

4. AI 能力展示（2页）
   - 三大创新点详解
   - 技术实现说明

5. 演示效果（1页）
   - 关键截图、数据表现
