## 模块一：用户洞察与问题定义

### 目标用户

**核心用户**：高校学生。

以北航的亲身经历为切入点，深入分析了浙大紫金港、北航学院路等多个校区的食堂场景后，我发现高校学生普遍面临相似的饮食决策困境——无论哪个学校，食堂窗口多、课业紧、营养信息缺。产品从我的真实就餐感受中泛化而来，服务于全体高校学生。

学生群体的饮食决策呈现高频次、短时间、强重复的特点——每周需做出 21 次用餐选择。

### 用户痛点

从就餐体验出发，我归纳出三个核心痛点：

**1. 选择困难**

每天站在食堂门口，面对多个楼层、数十个档口，纠结 10 分钟以上不知道该吃什么。选择太多反而无法选择，最后要么随便选然后后悔，要么花大量时间反复比较。在课业压力下，每多纠结 5 分钟就意味着少 5 分钟午休。

**2. 营养追踪与就餐场景割裂**

市面上不缺营养记录软件，但它们和实际就餐场景是分离的——吃完饭要打开另一个 app、手动搜索菜品、填写份量，操作成本太高，连坚持三天都难。真正的需求是"吃的时候顺手就记了"，而不是吃完再补一笔账。更进一步，如果能拍一餐饭自动识别菜品和价格，甚至可以顺手记账。

**3. 食堂信息停留在线下**

菜单更新、新菜推出、菜品评价仍然停留在纸质海报和食堂电子屏上。学生想知道"今天风味有什么新菜"、"麦香园的葱油拌面评价如何"，没有线上渠道可以及时了解。好吃不好吃全靠口口相传，信息流通效率极低。

### 使用场景

**场景一：午间速决**（高频，占 60%）

上午满课，12:05 下课，下午 13:15 有课。打开助手 → 说"帮我选一份午饭" → AI 结合偏好的"少油高蛋白"和当前菜单推荐 3 道菜 → 点击卡片查看详情 → 去对应窗口。全程不到 30 秒。

**场景二：饮食自律管理**（中频，占 25%）

正在减脂期，每天需要控制热量、保证蛋白质。打开助手 → 查看菜品详情中的营养成分 → 选择清蒸鱼 + 蔬菜沙拉 → 顺手记录到今日饮食 → 晚上查看今日汇总，确认摄入达标。**未来版本**：拍一餐饭的照片，AI 自动识别菜品、计算营养、完成记录和记账，把记录成本降到零。

**场景三：发现与分享**（低频，占 15%）

想尝试新东西。打开助手 → 浏览食堂列表 → 查看评分最高的菜品 → 看其他学生的点评 → 决定去试试麻辣香锅。吃到好吃的 → 随手评分 + 写两句评价 → 帮后面的同学做参考。

## 模块二：产品方案设计

### 产品概述

智能校园食堂助手是一个 AI 原生的校园生活工具，帮助学生快速决策"今天吃什么"，提供个性化推荐，并搭建营养追踪闭环。产品以"对话式交互"为核心交互范式，用户可以用自然语言表达需求，AI 理解意图后结合实时菜单数据给出推荐。

> **架构说明**：当前实现为 Web SPA，架构设计上预留了 QQ 小程序迁移能力。如后续落地 QQ 生态，前端 UI 层可复用现有组件，只需替换路由和登录层。当前 demo 中分享链路未实现。

> **演示说明**：除在线 Demo 外，随作品提交了产品交互演示动画（HTML），涵盖 AI 智能推荐、对话式交互、菜品详情与营养查看、UGC 菜单共建等核心模块的操作流程展示。

### 核心功能

| 功能 | 描述 | 当前状态 |
|------|------|---------|
| 食堂菜单浏览 | 查看各食堂菜品列表，支持按名称搜索 | 已实现 |
| AI 对话推荐 | 自然语言输入 → AI 理解 → 推荐菜品并附理由 | 已实现（DeepSeek API） |
| 偏好记忆系统 | 记录用户口味、预算、营养目标，持续优化推荐 | 已实现 |
| 菜品详情 | 查看营养数据（热量/蛋白质/脂肪）、用户评分 | 已实现 |
| 营养追踪打卡 | 手动记录每餐摄入，查看今日/累计营养汇总 | 已实现 |
| UGC 菜品点评 | 评分 + 文字评价，丰富菜品信息 | 已实现 |
| 收藏夹 | 收藏喜欢的菜品，方便快速访问 | 已实现 |
| 食堂地图 | 基于 Leaflet 的地图指引 | 已实现 |

### 产品愿景（后续迭代）

以下为产品的长期演进方向，当前 demo 未实现：

- **AI 拍照识别记录**：拍一餐饭的照片 → AI 识别菜品 → 自动计算营养和价格 → 一次性完成记录 + 记账。将"吃"和"记"在同一个动作里完成，把记录成本降到零。
- **QQ 生态集成**：分享推荐到 QQ 群、找饭搭子匹配。架构已预留，待落地时实现。

### 交互流程

**核心交互：AI 对话推荐**

```
用户打开 AI 推荐页
    ↓
AI 主动问候："今天想吃点什么？"
    ↓
用户输入："想吃辣的，预算 15 以内"
    ↓
前端构造请求：菜单预览 + 用户偏好 + 对话历史 → POST /api/chat
    ↓
后端调用 DeepSeek API → 解析推荐菜品 ID → 返回回复文本 + 菜品列表
    ↓
前端渲染：AI 回复气泡 + 菜品横向滚动卡片
    ↓
用户点击菜品卡片 → 跳转详情页查看营养数据
    ↓
用户可继续追问："有没有蛋白质更高的？"
    ↓
AI 结合上下文重新筛选推荐
```

**降级策略**：当 DeepSeek API 不可用时，前端启用本地多因子评分算法，基于标签匹配、营养计算、热度、价格、多样性、时间六个维度综合打分，确保核心推荐功能始终可用。

### 创新与差异化

**1. AI 对话式推荐 vs 传统表单式筛选**

现有食堂工具（如"浙大钉"）仅提供静态信息展示。我们的产品将交互从"选择筛选条件"升级为"自然语言对话"——用户不需要学习如何使用，说出需求即可。

**2. 偏好记忆系统**

AI 自动记忆用户的口味、预算、营养目标、就餐时间，不需要每次重新填写。记忆存储在 localStorage，用户可以随时开关或清除，兼顾便利与隐私控制。

**3. 本地+云端双引擎**

云端使用 DeepSeek API 进行智能对话与推理；断网或 API 故障时自动降级为本地多因子评分引擎（6 维加权：口味 30%、营养 25%、热度 20%、价格 15%、多样性 5%、时间 5%），保证核心功能不中断。

**4. 渐进式偏好收集**

当用户需求模糊时（如只说"帮我选一份午饭"），AI 通过多轮对话渐进式收集信息：先问口味 → 再问预算 → 再问营养目标，而非一次性抛出一个长表单，降低交互门槛。

## 模块三：AI 原生能力说明

### AI 核心能力

产品具备三项 AI 原生能力：

**1. 自然语言理解与对话推荐**

用户可以用自然语言描述饮食需求（"想吃辣的但不要太多油"、"今天想吃清淡的，最近在减脂"），AI 理解语义后，结合实时菜单数据推荐合适的菜品。

**2. 意图识别与渐进式交互**

当用户输入模糊（"推荐"、"随便"、空泛描述），AI 自动识别为"需求不明确"状态，主动追问具体偏好。系统预设了三级追问阶梯：口味偏好 → 预算范围 → 营养目标，每轮只问一个问题，避免信息过载。

**3. 多因子推荐评分**

本地引擎从六个维度计算菜品匹配度：
- **口味匹配（30%）**：菜品标签与用户口味偏好的重合度
- **营养匹配（25%）**：基于减脂/增肌/均衡三种目标，对热量、蛋白质、脂肪进行评分
- **热度评分（20%）**：基于用户评分和评价数量
- **价格匹配（15%）**：与用户预算的吻合度
- **多样性（5%）**：避免重复推荐同一菜品
- **时间匹配（5%）**：根据时间段（早餐/午餐/晚餐）调整推荐

### AI 如何解决痛点

| 痛点 | AI 方案 | 效果 |
|------|---------|------|
| 选择困难 | 对话式推荐，理解自然语言需求，10 秒给出 3 道推荐菜 | 决策时间从 10 分钟→10 秒 |
| 营养追踪断链 | 就餐场景内直接记录营养，AI 推荐后一键计入；未来拍照即识别 | 记录成本降到零，坚持门槛消失 |
| 食堂信息线下化 | AI 汇总评价和评分，热门/高分菜品主动呈现；UGC 评价持续沉淀 | 信息从"口口相传"变为"在线可查" |

**AI 原生性检验**：如果去掉 AI 能力，产品退化为一个静态菜单浏览工具——用户仍可查看菜品、浏览食堂信息，但核心价值（快速决策、智能推荐、个性化体验）完全消失。AI 不是点缀，而是产品的核心交互范式。

### AI 技术方案

```
用户输入 "推荐一个高蛋白的午饭"
    ↓
前端构建 system prompt（包含今日菜单预览 + 用户偏好记忆 + 对话历史）
    ↓
POST /api/chat → FastAPI 路由层
    ↓
请求 DeepSeek API（deepseek-chat 模型）
    ↓
AI 分析菜单 → 匹配用户需求 → 生成推荐文本
回复格式：推荐文本 + [DISHES:id1,id2,id3]
    ↓
后端解析 dish IDs → 返回结构化数据 {reply, dishes, quick_reply?, context?}
    ↓
前端渲染推荐结果 → 用户点击查看详情
```

**技术要点**：
- **模型**：DeepSeek-chat，5-10 秒超时，低 token 消耗（50-200 tokens/次）
- **Prompt 设计**：system prompt 包含全部菜品的结构化摘要（名称、价格、热量、标签），使 AI 具备"知晓今日菜单"的能力
- **上下文管理**：维持最近 4 轮对话历史，支持追问和修正
- **偏好注入**：用户的历史偏好作为系统指令的一部分传入，AI 不需要重新询问已知信息
- **降级策略**：API 不可用时，前端本地引擎接管（见模块二多因子评分）

**API 端点**：
- `POST /api/chat` — AI 对话推荐（主接口）
- `POST /api/recommendations/ai` — 批量推荐 + 理由生成（辅助接口）
- 共计 24 个 RESTful API 端点

## 模块四：加分项

### 落地可行性

**技术可行性**：
- 前端使用 React 19 + Vite 6，部署成本极低（静态文件托管或 CDN）
- 后端 FastAPI，单体架构，单服务器可承载千级并发
- 数据层当前为内存存储，可平滑迁移至 SQLite/PostgreSQL
- 唯一外部依赖为 DeepSeek API，无其他基础设施成本
- Demo 模式下 48 小时内从零完成开发验证了技术栈的可行性

**数据获取路径**（三阶段）：
1. 冷启动：人工录入高频菜品（已录入 32 道菜，覆盖 3 个食堂）
2. UGC 众包：学生上传菜单照片 + 点评，平台审核后入库
3. 校方合作：对接食堂管理系统，获取实时菜单和营养数据

**推广路径**：
- 初期：QQ 群 + 朋友圈分享裂变
- 中期：食堂门口地推 + 学生组织合作
- 长期：校方合作，纳入官方服务

### 商业化思考

**价值网络三方平衡**：

| 角色 | 价值 | 商业模式 |
|------|------|---------|
| 学生（用户） | 决策提效、营养管理、美食发现 | 免费使用 |
| 食堂（商家） | 获得流量分发、用户反馈数据 | 菜品推广位、优惠券分发（按核销收费） |
| 平台（我们） | 连接供需、数据沉淀 | 抽成 + 数据服务 + 校企合作经费 |

**具体商业路径**：
1. **食堂推广位**：新菜上市、特价菜在推荐结果中获得优先展示，食堂按 CPC/CPM 付费
2. **优惠券分发**：食堂发放电子优惠券，平台按核销量抽佣（参考美团模式）
3. **数据服务**：向校方后勤提供学生饮食偏好分析报告（匿名化），帮助优化食堂菜品配置
4. **校企合作**：作为智慧校园建设的一部分，由学校采购服务

**规模化潜力**：
- 横向扩展：从浙大紫金港扩展到其他校区、其他高校
- 纵向扩展：从食堂扩展到校外周边餐饮
- 已验证的 48 小时开发周期表明，新产品校区冷启动的技术成本极低
